# 2 Zombie 框架与随机游走算法分析

## 2.1 Zombie 框架概述与求解流程

Zombie 是一个面向椭圆型与抛物型偏微分方程的随机数值求解框架，其核心思想基于 Monte Carlo 随机游走方法，通过构造大量独立随机轨迹对 PDE 解的边界贡献进行统计估计。与传统基于网格离散的数值方法不同，Zombie 属于典型的无网格（meshfree）求解体系，其计算过程不依赖全局网格剖分，而是通过局部几何查询与随机采样实现对解的近似计算。

Zombie 框架主要支持 Walk-on-Spheres（WoS）、Walk-on-Stars（WoSt）以及 Boundary Value Caching（BVC）等随机游走算法变体，可用于求解 Poisson 与 screened Poisson 等典型边值问题。框架整体采用模块化设计，将几何描述、边界条件处理、随机采样、轨迹推进与统计归约等功能进行解耦，从而具备较好的算法扩展性与工程复用能力。

从系统结构角度看，Zombie 的求解过程由若干核心功能模块协同完成。其中，几何与边界模块负责描述计算域边界并提供随机游走过程中所需的距离查询能力；Green 函数与核函数模块定义了随机游走推进过程中的解析核形式，决定球半径更新与贡献累积的数值特征；随机采样模块负责生成球面方向等随机变量，是 Monte Carlo 随机性的主要来源；随机游走执行模块实现 WoS/WoSt 的核心推进循环；统计归约模块则对大量独立轨迹的终止贡献进行统计平均，最终输出 PDE 解的估计值及误差指标。

Zombie 采用模块化设计，将求解过程解耦为求解器/点估计（Solver / Point Estimator）、随机游走执行（Random Walk Execution）与若干基础支撑模块。求解器在给定 PDE 类型、边界条件与几何描述后，按采样点组织 Monte Carlo 估计并完成统计归约（均值/方差等）。随机游走执行层实现 WoS/WoSt 等算法的轨迹推进，其中 Random Walk Kernel 为主要计算热点，反复调用几何与边界查询（距离/交点等）、随机采样（RNG/方向采样）以及 PDE/解析核相关计算，并可与方差缩减模块（如 Boundary Value Caching，BVC）协同以降低方差。该结构表明 Zombie 的计算开销主要集中在随机游走内核及其对几何查询与采样模块的高频依赖上，为后续并行化与体系结构优化提供了明确的热点定位与模块边界。

在给定 PDE 类型、边界条件与几何描述后，Zombie 的典型求解过程表现为对多个采样点分别执行大量随机游走轨迹，并通过统计归约获得数值解估计。求解器首先在计算域内部生成待求解的采样点集合，每个采样点对应 PDE 解的一个估计位置。随后，对于每个采样点，系统独立执行多条随机游走轨迹。随机游走从初始位置出发，在迭代推进过程中反复进行边界距离计算、可行球半径更新、随机方向采样以及位置推进，并在满足吸收边界命中或收敛阈值条件时终止。轨迹终止后，根据边界条件计算对应的终止贡献，并对所有轨迹结果进行统计平均，从而得到采样点处 PDE 解的 Monte Carlo 估计值。

通过上述执行流程可以看出，Zombie 框架的主要计算开销并不来源于传统网格方法中的全局线性代数求解，而是集中于大量随机游走轨迹的推进执行过程。由于随机游走轨迹之间相互独立，Zombie 在任务层面具有天然的点级并行性。然而，单条随机游走的执行步数具有显著随机性，不同轨迹的计算成本高度不均衡，同时推进过程中包含复杂的终止判定与控制流分支，使得其执行模式呈现强不规则特征。

因此，Zombie 的并行化与体系结构迁移的关键挑战并非并行性是否存在，而在于如何在众核体系结构上高效组织这些执行成本不可预测的随机游走任务，避免负载失衡并充分发挥硬件吞吐能力。上述分析为后续章节中 Walk-on-Spheres 随机游走算法的执行机制解析以及动态粒度调度与混合精度优化方法的提出奠定了基础。

## 2.2 Walk-on-Spheres 随机游走算法执行机制

Walk-on-Spheres（WoS）方法是 Zombie 框架中求解椭圆型边值问题的核心随机游走算法之一，其基本思想是利用调和函数的平均值性质，通过在计算域内部构造一系列球跳跃过程，将 PDE 解的求解转化为边界贡献的统计估计。与传统基于网格离散的数值方法相比，WoS 方法避免了全局网格剖分与线性方程组求解，而是通过局部几何查询与随机采样实现对解的无网格近似。

在 WoS 方法中，随机游走轨迹从域内采样点出发，在每一次迭代中根据当前位置到边界的最小距离构造一个完全包含于计算域内部的最大内切球。随后，轨迹在该球面上随机采样一个方向，并跳跃至球面上的新位置。该过程不断重复，直到轨迹到达吸收边界附近或满足预设终止条件。轨迹终止后，其边界命中位置对应的边界条件值将作为该条随机游走的贡献项，并通过对大量轨迹结果的统计平均得到采样点处 PDE 解的估计值。

WoS 方法的执行过程具有典型的“连续推进—终止吸收—贡献累积”结构，其核心计算由随机游走主循环驱动。为了明确其在 Zombie 框架中的实现形式，本节给出 WoS 随机游走的典型推进伪代码。

从代码执行角度看，WoS 随机游走的每一次迭代均包含若干固定的计算阶段。首先需要进行几何查询，以确定当前位置到边界的最近距离并构造最大内切球；随后根据 Green 函数形式更新推进参数，并在球面上完成随机方向采样；接着轨迹位置被推进至新的球面点，同时累积可能的贡献项；最后需要进行终止判定，判断轨迹是否已经到达吸收边界或进入收敛阈值范围。上述阶段在每条随机游走轨迹中反复执行，构成 Zombie 框架的主要计算热点。

需要指出的是，WoS 方法虽然在任务层面具有天然的独立性，但单条轨迹的执行步数与终止位置均具有显著随机性。在实际求解过程中，不同轨迹可能在极少数迭代后快速终止，也可能经历大量球跳跃后才到达边界附近，从而导致轨迹之间的计算成本呈现高度离散分布。此外，推进过程中频繁的几何距离查询与终止分支判断使得其控制流表现出明显的不规则特征。这些特性决定了 WoS 随机游走任务难以通过传统静态划分实现高效并行，尤其在众核体系结构上容易产生严重的负载失衡与资源利用率下降。

因此，在 Zombie 框架的并行化与体系结构迁移过程中，WoS 随机游走轨迹通常被视为最小的任务执行单元，其动态生成、执行与终止行为构成了整体计算负载的基本形态。对该执行机制的深入分析不仅揭示了 Zombie 的核心计算特征，也为后续章节中随机游走任务抽象、动态粒度调度策略以及阶段感知混合精度优化方法的提出提供了直接依据。

## 2.3 随机游走任务的不规则性来源分析

尽管 Walk-on-Spheres 方法在轨迹层面具有天然的独立性，为并行化提供了理论上的任务并行基础，但其实际执行过程呈现出显著的不规则计算特征。这种不规则性主要来源于随机游走轨迹推进行为的高度随机性、控制流结构的复杂性以及动态任务生成机制的不可预测性。上述因素共同导致 Zombie 框架在众核体系结构上的并行效率难以通过传统静态划分策略获得保障，负载失衡与资源利用率下降成为其性能优化的核心瓶颈。

首先，随机游走轨迹的迭代步数具有显著的随机离散性。在 WoS 推进过程中，轨迹每次跳跃的球半径由当前位置到边界的距离决定，而不同轨迹在域内的空间演化路径完全由随机采样控制，因此其终止位置与终止时间均不可预测。在实际求解中，部分轨迹可能在极少数迭代后迅速命中边界并终止，而另一些轨迹则可能经历大量球跳跃才进入吸收区域，从而表现出明显的长尾分布特征。这意味着即便为每个线程分配相同数量的轨迹任务，不同线程的实际计算量仍可能存在数量级差异。

随机游走推进过程包含大量终止判定与分支控制逻辑，使得其控制流呈现显著的不规则性。在每次迭代中，轨迹不仅需要进行几何距离查询与球半径更新，还必须判断是否满足吸收边界命中、收敛阈值触发或概率截断等终止条件。由于不同轨迹在空间位置与推进阶段上高度异步，这些分支行为在并行执行时难以保持一致，从而导致线程间控制流发散与执行效率下降。特别是在众核体系结构中，大量轻量线程的分支不一致会进一步放大硬件资源浪费。

Zombie 框架在实际求解中往往结合多种方差缩减策略，例如轨迹分裂（splitting）、轮盘赌截断（roulette）或边界缓存机制等。这些策略虽然能够改善 Monte Carlo 估计的收敛效率，但同时引入了动态任务队列膨胀现象，即随机游走任务在执行过程中可能生成新的子任务或产生额外的状态分支，从而导致任务数量与执行顺序在运行时不断变化。这种动态性进一步削弱了静态任务划分的有效性，使得并行调度必须具备运行时自适应能力。

因此，在将 Zombie 框架迁移至神威等国产众核体系结构时，必须引入面向不规则任务的动态粒度调度机制，通过运行时任务重分配与负载自适应策略缓解长尾效应，并提升海量计算核心的资源利用率。

## 2.4 神威众核体系下的执行挑战与迁移需求

在前述分析中可以看到，Zombie 框架的核心负载由大量随机游走轨迹推进任务构成，其执行步数、终止行为与控制流结构均具有显著随机性。这类负载在传统 CPU 平台上通常依赖缓存层次与线程级调度机制获得一定的容错空间，但在神威众核体系结构下，上述不规则特征会直接转化为体系结构适配问题，成为限制并行效率的主要因素。

神威新一代众核处理器采用主核（MPE）与从核（CPE）协同的执行模式。从核计算单元具备较高的并行吞吐能力，但其性能发挥高度依赖局部数据存储（LDM）的显式管理。LDM 容量有限且不提供自动缓存一致性支持，数据需要通过 DMA 显式搬运进入局部空间后才能高效访问。因此，随机游走推进过程中涉及的轨迹状态更新、边界几何查询与核函数参数访问，若仍沿用面向 CPU 缓存的存储组织方式，将导致频繁的非连续访存与数据搬运开销，难以形成稳定的计算密度。

另一方面，Zombie 的随机游走任务具有明显的执行成本离散性。不同轨迹的迭代次数差异较大，长尾现象普遍存在。在海量从核并行环境下，静态划分策略会导致部分计算核心提前完成而进入空闲状态，而少数核心仍需处理高成本轨迹，从而造成全局同步点上的等待开销。该类负载失衡并非实现细节所致，而是随机游走算法本身的内在属性，在众核规模扩展后会进一步放大。

随机游走推进过程中的控制流特征同样对神威体系结构提出约束。每次迭代包含终止判定、边界吸收处理以及可能的分裂与截断操作，分支行为随轨迹状态动态变化。由于不同轨迹在推进阶段上高度异步，线程间控制流难以保持一致，从核 SIMD 执行单元的利用率容易下降。同时，动态任务队列的膨胀使得任务粒度与任务数量在运行时持续变化，对主核侧的调度开销与任务管理机制提出了额外压力。

综上，Zombie 框架在神威平台上的迁移并不属于直接的并行接口替换问题，而需要围绕随机游走任务的组织方式进行重构。一方面需要引入面向不规则负载的动态粒度调度机制，以缓解轨迹执行长尾带来的负载失衡；另一方面需要结合 LDM 管理与 DMA 搬运约束，对轨迹状态与几何数据访问模式进行重组，降低访存开销并提高从核计算效率。此外，随机游走推进过程在不同阶段对数值精度的敏感性存在差异，为后续阶段感知的混合精度优化提供了实现基础。

## 2.6 本章小结

本章围绕 Zombie 框架的随机游走求解流程与计算特征展开分析。首先对框架的模块组成与典型执行过程进行了概述，明确了其主要计算负载集中在随机游走轨迹的推进阶段。随后对 Walk-on-Spheres 方法的执行机制进行了细化讨论，给出了单条轨迹在几何查询、随机采样、位置推进与终止吸收等环节中的迭代结构。

本章从轨迹步数离散性、控制流分支复杂性以及动态任务生成行为等方面分析了随机游走任务的不规则性来源。研究表明，随机游走轨迹的执行成本具有显著随机性与长尾特征，静态划分策略难以避免负载失衡，众核并行环境下资源利用率容易受到限制。

最后结合神威众核体系结构的主从协同模式与显式局部存储管理机制，讨论了 Zombie 框架迁移过程中面临的执行约束。随机游走任务的动态性与不规则访存模式对任务组织方式与数据布局提出了体系结构导向的重构需求。

基于本章分析，后续章节将进一步建立 Zombie 在神威平台上的映射模型，并在此基础上分别展开面向不规则随机游走任务的动态粒度调度策略与阶段感知混合精度优化方法研究，以提升国产众核体系结构上的并行效率与计算性能。
