# 6 Zombie 迁移与原型系统实现

## 6.1 原型系统架构与迁移改造总览

本章面向神威 SW26010P 平台实现 Zombie 随机游走求解流程的并行化原型系统，并将第4章的动态粒度调度与第5章的阶段感知混合精度以最小侵入方式集成到核心执行路径中。第6章的写作目标是给出可复现的系统结构、明确的改造边界与稳定的测量接口，为第7章实验对比提供工程基础。

### 6.1.1 迁移目标与设计约束

原型系统的迁移目标包括三个方面。第一，将 Zombie 的随机游走主路径组织为任务块执行模型，使大量独立轨迹以任务块 WalkTask 为调度与执行单元，在 MPE–CPE 协同模式下高吞吐运行。第二，将动态粒度调度机制落实到任务块边界，使用粒度参数 B 控制任务块所包含的轨迹数量，并在尾部阶段通过回收与再分配降低负载失衡程度。第三，将阶段感知混合精度策略落实到 WalkTask::execute 内部，以阶段标志 p∈{P0,P1,P2,P3,P4} 驱动精度模式 s∈{FP32,FP64} 的选择与切换，保证离散事件判定与贡献计算处于 FP64 保护之下，同时将常规步进阶段置于 FP32。

为保证可对照实验与可复现实现，原型系统遵循如下设计约束。系统在调度层保持任务语义稳定：任务块在队列 Q 中的表示不随混合精度策略变化，任务的提交、拆分、回收与分发流程保持一致。混合精度仅作用于执行层：所有精度选择与切换仅发生在 WalkTask::execute 内部的阶段边界处，不进入队列操作与任务封装路径。系统在数值语义层保持求解形式不变：随机游走估计方式、终止贡献计算方式与边界条件处理方式保持与原始 Zombie 一致，仅改变部分算子的浮点精度实现。系统在测量接口层提供固定口径：吞吐 θ、加速比 S、效率 E、尾部比例 与相对误差 的统计口径在系统层固化，便于第7章直接引用。

### 6.1.2 原型系统总体架构

原型系统采用典型的管理核（MPE）与计算核组（CPE）协同结构。MPE 负责全局流程控制、任务块生成与队列 Q 维护，以及结果归约与输出；CPE 负责从队列获取任务块并执行 WalkTask::execute，在本地完成轨迹推进与贡献累积，再将局部结果回写到共享内存并通知 MPE 归约。队列 Q 以任务块为元素，任务块携带粒度参数 B 与执行所需的参数集合；混合精度相关策略参数（,,）由 MPE 下发并在 execute 内读取，精度模式 s 在执行端按阶段标志 p 动态确定。

如图6-1所示，系统的数据通路可分为三类：几何数据通路、轨迹状态通路与统计通路。几何数据通路包含边界几何与加速结构，用于计算 d(x) 与近边界几何量；轨迹状态通路包含 WalkState 的集合与任务块级缓冲，用于维护 t,x,w,k,term,rng 等状态并在任务块边界回写；统计通路输出任务执行计数器与时间信息，用于构造 θ,E,等指标并进行开销分解。

### 6.1.3 最小侵入改造边界与对照组一致性

原型系统对 Zombie 的改造集中在随机游走主路径的执行组织方式上，改造目标是将“逐轨迹执行”重构为“任务块执行”，并在任务块边界插入调度与统计接口。改造不改变随机游走估计的数学形式，不改变边界条件贡献的计算方式，不改变终止判定条件集合，仅将执行流程划分为可调度的任务块，并在执行端引入阶段标志 𝑝p 与精度模式 s 的控制逻辑。

为保证第7章的对照实验成立，系统提供稳定的版本开关与配置接口，使不同策略在同一代码路径下仅通过参数与开关差异实现。对照组定义遵循“同一任务块模型、同一队列模型、同一数据布局”的原则：基线版本关闭动态粒度回收与混合精度，执行端以 FP64 覆盖全部阶段；仅启用动态粒度版本保持全 FP64，启用回收与再分配；仅启用混合精度版本保持固定 B，在 execute 内按 s(p) 切换精度；联合版本同时启用动态粒度与混合精度，并形成配置 (B,s) 的组合集合用于性能—精度权衡评估。上述对照组的参数集合与开关列表在 6.4 中以表格形式固化。

为了支撑后续实验的开销分解，本章在系统层统一定义任务块生命周期的时间戳与计数器采集点，并在 CPE 端对阶段执行次数、精度切换次数与阶段耗时进行统计。统计数据的采集位置与统计方法在 6.4 给出，作为第7章实验指标的直接来源。

## 6.2 任务块执行链路与动态粒度调度落地

本节给出任务块执行链路的系统实现，明确 WalkTask 的语义边界、队列 Q 的组织方式与任务块生命周期，并在该执行链路上落地第4章的动态粒度调度机制。该部分内容决定调度开销的上界与负载均衡能力的上界，也是第7章尾部比例 、效率E 与吞吐 θ 对比实验的实现基础。

### 6.2.1 任务块 WalkTask 的系统定义与状态组织

原型系统以任务块 WalkTask 作为最小调度单元。每个任务块包含 B 条独立随机游走轨迹的状态集合，B 为任务块粒度参数，与第4章一致。任务块内部的每条轨迹以 WalkState 表示，其状态量包含 x,w,k,term,rng，并在 WalkTask::execute 内按第5章阶段标志 p 推进。

在系统实现中，WalkTask 的输入由两类信息组成：轨迹初始状态与执行配置。轨迹初始状态由采样点与边界条件配置生成，进入任务块时被封装为 WalkState 数组；执行配置包含 B、最大步数 、近边界阈值 以及混合精度策略的阈值参数 ,。执行配置作为只读参数下发，任务块执行过程中不修改该配置，确保任务块在队列 Q 中的表示稳定。

任务块内部采用“批处理+局部归约”的组织方式。每条轨迹在推进过程中产生的贡献项先累积到任务块本地缓冲，再在任务块结束时一次性回写到共享内存并触发 MPE 归约。该设计将高频的小粒度写操作收敛到任务块边界，降低共享内存写回压力，并为尾部回收机制提供天然的切分点。

WalkTask 在系统中经历提交、分发、执行、回写以及完成/回收等固定阶段。动态粒度调度仅改变任务块内部轨迹集合规模与尾部回收时机，不改变任务块的生命周期结构。其中，局部贡献在 execute 阶段先累积到任务块本地缓冲，并在 commit 阶段集中回写；当尾部回收条件满足时，未完成轨迹被重新封装并再次入队，否则任务块完成并退出队列。

### 6.2.2 队列 Q 与 MPE–CPE 分发模型

队列 Q 维护待执行任务块集合，由 MPE 负责入队与全局终止判定，由 CPE 负责出队与执行。队列元素为 WalkTask 句柄或等价描述符，包含指向轨迹状态数组的地址、任务块粒度 B、执行配置参数指针、以及任务块级统计计数器的存储位置。队列操作以任务块为原子单位，避免在调度层引入更细粒度的同步。

MPE 侧的任务生成流程按采样点或采样点批次进行：对每个采样点生成初始轨迹集合，将其按 B 划分为若干任务块并入队。CPE 侧采用“空闲即取”的拉取模式，从队列获取任务块后执行 WalkTask::execute，并在任务块结束时回写局部贡献与统计信息。MPE 在收到任务块完成通知后执行归约与全局结束判定。

该分发模型的同步边界固定在任务块边界处，调度层不介入 execute 内部阶段推进。该约束保证第5章的精度切换仅发生在执行路径内部，不与队列语义耦合，也保证第4章的回收机制在任务块边界实施，不破坏轨迹推进的局部一致性。

为便于复现实验与开销分解，系统在队列操作处记录时间戳与计数器，包括队列等待时间、任务块执行时间与回写归约时间。相关统计量的定义与采集点在本节末的表6-2固化，并在第7章作为调度开销与尾部分析的直接依据。

### 6.2.3 动态粒度调度与尾部回收的系统实现

动态粒度调度以“稳态粗粒度、尾部细粒度”为执行原则。系统在稳态阶段使用较大的 B 降低调度频率并提高数据复用；当进入尾部阶段时，系统通过回收机制将尚未完成的轨迹从长任务块中抽取出来，重封装为更小粒度的任务块重新入队，从而缩短尾部等待时间并降低 。

尾部阶段的判定在 MPE 侧基于队列状态与已完成任务块比例实现，回收触发在 CPE 侧基于任务块内部轨迹完成比例实现。CPE 在 execute 返回时检查任务块内轨迹的完成标志 term，对未完成轨迹集合进行压缩打包；若未完成轨迹数量低于回收阈值，则触发回收，将其封装为新的任务块并入队。该过程不修改任何已完成轨迹的贡献累积结果，仅改变未完成轨迹的后续调度方式，满足“调度语义不改变求解语义”的要求。

动态粒度调度的系统实现以两段伪代码描述：MPE 侧任务生成与队列管理流程见算法6-1，CPE 侧取任务、执行、回写与回收流程见算法6-2。两段伪代码共同确定任务块生命周期、回收触发点与回收重封装规则，构成第4章算法在系统中的落点。

回收判断 NeedRecycle(exec_stats) 的输入口径在系统中固定，包括任务块内未终止轨迹数量、任务块执行时间与尾部判定标志。回收重封装 RepackUnfinishedStates 仅对未完成轨迹集合做压缩与重新分块，不改变随机数状态 rng 与轨迹状态量 x,w,k，保证回收前后轨迹推进的连续性。

为支撑第7章对调度开销与尾部缓解效果的量化分析，系统在 MPE 提交与接收阶段、CPE 取任务与提交阶段以及任务块回收点记录统一的调度统计信息。具体而言，任务块执行时间在 CPE 侧以 fetch–commit 区间计量，用于计算吞吐  与并行效率 ；尾部任务块数量及其占比在回收判定与重封装阶段统计，用于构造尾部比例 ；队列提交、接收与回写相关计数则用于调度开销分解。由于回收重封装仅压缩未完成轨迹集合而不改变随机数状态与轨迹状态量，因此统计口径在任务块回收前后保持一致。

## 6.3 任务块执行链路与动态粒度调度落地

本节给出阶段感知混合精度在原型系统中的落地方式，重点说明 WalkTask::execute 的阶段化执行骨架、精度映射 s=s(p) 的实现位置与实现口径，以及与调度层解耦的工程约束。该部分内容对应第5章的方法定义，决定第7章“仅混合精度”和“联合机制”的性能收益与误差控制结果是否可复现。

### 6.3.1 WalkTask::execute 的阶段化执行骨架

WalkTask::execute 是混合精度策略在系统中的唯一作用位置。执行函数以任务块为单位推进轨迹集合，对每条轨迹按第5章给出的阶段集合p∈{P0,P1,P2,P3,P4}进行阶段判定与阶段转移，并在 P4 完成终止判定与终止贡献累积。阶段化执行骨架固定为“几何评估—步进更新—近边界处理—终止与贡献”的循环结构：轨迹在 P1计算 d(x) 并决定是否进入近边界处理；在 P2 完成随机采样与位置更新；当 d(x)≤ 时进入 P3完成近边界几何精细计算，随后进入 P4完成终止与分支判定以及贡献累积。该骨架与调度层无关，不读取队列 Q 状态，不读取任务块回收标志，仅依赖轨迹局部状态 (x,w,k,term,rng) 与只读阈值参数。

为避免执行端产生跨轨迹的同步开销，任务块内部采用“轨迹独立推进 + 任务块内局部归约”的实现方式。每条轨迹在本地寄存器与局部缓冲中维护状态，阶段切换仅发生在该轨迹的阶段边界处；贡献项累积写入任务块局部缓冲，任务块结束时一次性归约回写。该设计将高频写操作收敛到任务块边界，与 6.2 的调度边界保持一致。

WalkTask::execute 的阶段化执行骨架与精度切换位置沿用图5-5所示定义；系统实现仅将该阶段规则嵌入任务块执行内核，并与任务块内局部归约机制结合。

### 6.3.2 精度映射 s(p) 的实现口径与参数下发

精度映射采用第5章的形式

s=s(p),s∈{FP32,FP64}.

系统默认采用固定映射：P2使用 FP32，P1,P3,P4使用 FP64。当启用阈值滞回策略时，系统在 execute 内维护近边界保护区标志 z，并在 P1阶段按 z 选择距离计算的精度模式；阈值参数 ,由 MPE 在任务块配置中下发，CPE 端作为只读参数使用。近边界触发阈值 与最大步数 同样作为只读参数下发，并在阶段判定与终止判定中使用。参数下发采用“任务块配置指针”的方式实现，避免在队列元素中复制大块参数，保证队列 Q 的操作成本稳定。

精度映射在实现上对应到“算子级精度版本选择”。系统将与事件判定相关的算子固定为 FP64 版本，包括几何距离与近边界判定、近边界几何量计算、终止与分支判定以及终止贡献计算；将常规步进阶段的采样与位置更新固定为 FP32 版本。该划分与 5.3 的误差传播结论一致，使事件敏感阶段处于高精度保护之下，步进更新阶段以低精度降低单位轨迹成本。

### 6.3.3 与调度层解耦的实现边界与统计口径

混合精度的实现边界在系统层被严格限制在 execute 内部。队列 Q 的任务元素、任务块拆分与回收逻辑、回写与归约逻辑不携带“当前精度状态”，也不依赖阶段标志 p 的瞬时取值。任务块在队列中的表示包含粒度参数 B 与策略参数指针，但不包含运行中产生的阶段统计信息；运行中统计信息在任务块结束时作为回写消息的一部分发送给 MPE，用于实验开销分析，不参与调度决策。该实现方式确保启用或关闭混合精度不会改变队列行为，从而保证调度语义在不同策略之间一致。

为支撑第7章的精度—性能权衡分析，系统在 execute 内记录混合精度相关统计量，并在任务块结束时回写。统计量包括阶段执行次数、精度切换次数、P1中 FP32/FP64 的使用次数（启用滞回策略时）、以及阶段耗时累计。统计信息的采集点固定在阶段入口与阶段出口，计时采用同一套硬件计数器口径，确保不同策略之间可比。统计口径在 6.4 汇总为实验采集对照表，用于第7章开销分解与消融实验。

WalkTask::execute 内部的阶段化混合精度执行规则沿用算法5-1；在系统实现中，仅将其作用域扩展到任务块级执行框架，即对任务块内各轨迹独立调用相同的阶段推进逻辑，并将贡献项先累积到任务块局部缓冲，任务块结束时再统一回写。

## 6.4 联合配置 (B,s) 协同机制与可复现实验接口

本节在 6.2 的任务块与动态粒度调度实现、6.3 的阶段化混合精度实现基础上，给出联合配置 (B,s) 的系统协同机制，并固化第7章实验所需的配置集合与指标采集口径。该部分内容的目标是使“方法—系统—实验”形成闭环：系统端明确哪些开关与参数构成实验对比组，明确每个指标从何处采集、以何种统计方式计算，保证实验结果可复现、可解释。

### 6.4.1 联合配置 (B,s) 的系统表达与传递

原型系统以任务块为边界形成运行时联合配置。粒度参数 B 由任务块 WalkTask 的封装决定，影响每次队列交互携带的轨迹数量与调度频率；精度模式 s 由执行端在 WalkTask::execute 内按阶段标志 p 决定，形成 s=s(p)。为统一描述，本文将每个运行配置记为二元组(B,s),

其中 B 是任务块粒度，s 是由策略决定的阶段内精度选择规则。该二元组在系统实现中对应两部分：任务块头部携带 B 与指向配置结构的指针；配置结构包含 ,,,与混合精度策略开关，执行端在阶段入口处据此确定 s。联合配置的表达方式不改变队列 Q 的元素语义，不将阶段标志 p 或精度状态写入队列，保证调度语义稳定。

联合配置的协同机制体现在两个层面。其一，B 决定任务块执行时间分布，从而影响尾部比例 与效率 E；其二，s 决定单位轨迹的执行成本，从而影响吞吐θ 与能效相关开销。二者共同决定系统整体性能与误差控制边界，并在第7章通过消融与联合对比进行量化验证。

### 6.4.2 实验对比组与配置集合固化

为保证第7章对照实验成立，系统将策略开关与参数组合固化为有限配置集合。配置集合覆盖四类对比组：基线组、仅动态粒度组、仅混合精度组、联合组。四类对比组共享同一套任务块执行链路、同一套队列模型与数据布局，仅在调度回收开关、粒度参数 B 与混合精度策略开关上存在差异。

基线组关闭动态粒度回收与混合精度，执行端以 FP64 覆盖全部阶段，作为参考实现。仅动态粒度组启用回收与再分配，保持全 FP64 执行，用于隔离调度机制带来的收益。仅混合精度组保持固定 B，启用 s(p) 策略并关闭回收，用于隔离混合精度带来的收益与误差影响。联合组同时启用回收与混合精度，在相同的采样规模与任务生成规则下评估协同效果。

## 6.5 本章小结

本章围绕神威 SW26010P 平台完成 Zombie 随机游走求解流程的迁移与原型系统实现，并将第4章动态粒度调度与第5章阶段感知混合精度落实到同一条可复现的执行链路中。首先，本章给出原型系统的总体架构与改造边界：MPE 负责任务生成、队列 Q 管理与全局归约，CPE 负责任务块执行与局部贡献累积；混合精度的作用位置被严格限定在 WalkTask::execute 内部，调度层任务语义保持不变，为对照实验提供一致的系统基线。

随后，本章以任务块 WalkTask 作为最小调度单元，给出队列分发模型与任务块生命周期，实现第4章动态粒度调度的工程落点：通过任务块粒度参数 B 控制调度频率，并在尾部阶段对未完成轨迹执行回收、重封装与再入队，以降低尾部比例 并提升并行效率 E。在执行端，本章将第5章阶段集合 p∈{P0,P1,P2,P3,P4} 显式嵌入 execute 的阶段化执行骨架，并实现精度映射 s=s(p)：将步进更新阶段 P2置于 FP32，将几何评估、近边界处理与终止判定贡献计算相关阶段置于 FP64，并在启用滞回策略时引入 ,对 P1的距离计算精度进行细化控制。精度切换仅发生在阶段入口处，切换次数与阶段转移次数一致，便于开销统计与复现。

最后，本章以联合配置 (B,s) 统一描述运行时配置，固化第7章实验对比组与参数集合，并给出指标定义、采集位置与统计方法：性能指标 θ,S,E，负载指标 ，精度指标 ，以及调度与混合精度相关的开销统计口径均在系统层形成稳定接口。基于上述实现，第7章可在同一代码路径与一致任务语义下完成基线组、仅动态粒度组、仅混合精度组与联合组的对照实验，并进一步进行开销分解与消融分析。
